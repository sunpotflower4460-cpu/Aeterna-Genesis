# LAW.md — Aeterna-Genesis の掟（全モジュール厳守）

このファイルは、Aeterna-Genesis のすべてのモジュールが従う規則の全文である。
目的はただ一つ：**擬似（意図どおりに動かす作り物）と、本物の創発を、毎回
機械的に区別すること。**

> **【2026-07 改訂 — GPT 監査統合】** 二値の「GREEN／それ以外」は**廃止**し、
> 役割 **E / V / S / N / F / Q**（§6）に置き換える。以後「GREEN」とは
> **E か V** のことだけを指し、**7 監査（§2）＋第8監査（§7）を全通過**したものに限る。
> V（教科書・既知結果との数の一致）は E に**劣らない**——硬い検証である。
> 追加された三大原則（§9）と**第8監査**（評価ゲート・初期条件が結論と同型の因果を
> 符号化していないか）は、§0〜§5 と同格に全モジュールへ適用される。
> 詳細は §6〜§9。旧文中の「GREEN」は「E か V」と読み替える。

---

## 0. 「本物の物理」の二つの意味（混ぜると擬似に転ぶ）

- **(A) 忠実な創発**：本物の法則（実在の場の方程式）を入れ、最小条件から、
  現象が勝手に出る。法則は "与えられて本物"、結果は "勝手に出る"。
  例：グロス・ピタエフスキー方程式（実在の BEC/超流体の式）＋初期条件
  → 渦が動き回り合う。**これは本物の物理。今すぐ厳密にできる。**
- **(B) 法則自体を根から導く**：場の方程式さえ 0≠無 から出す。最深の目標。
  最前線。**まだ。**

このリポジトリは **(A) を厳密に積み、その上で (B) へ向かう。** どのモジュールも
「今これは (A) か (B) か、どの入力をまだ手で入れているか」を明記する。

---

## 1. 全実験モジュールの先頭ヘッダ（必須・例外なし）

各 `run.py` の先頭、および `AUDIT.md` に、この 10 項目を必ず書く：

```
MODULE:      e0NN_<name>
QUESTION:    何の創発を問うか（1文）
PUT IN:      入れたもの = 場の方程式/局所規則 ＋ 初期/境界条件。これ"だけ"を列挙
EMERGED:     出たもの = 入れていないのに出た構造
CLAIM TIER:  measured | observed | interpretive | analogy | frontier
KNOWN MATCH: 既知物理との一致（どの実在の結果と合うか＋文献）
AUDIT (7):   下の7項目それぞれに Yes/No と理由
STATUS:      GREEN(全通過) / YELLOW(注記つき) / RED(未達)
A_OR_B:      この実験は (A) か (B) か。まだ手で入れている入力は何か
```

---

## 2. 七つの監査（擬似 vs 本物を機械的に切り分ける）

各モジュールで必ず答える：

1. **規則は結果に言及していないか**：局所の量だけで書かれ、「渦を作れ／3次元
   にしろ」と結果を名指ししていないか。していれば台本＝擬似。
2. **忠実な物理か**：既知の正しい法則、または無理のない局所動力学か。
   **結果に合わせて作った "生成器" でないか。**
3. **結果は初期条件に入っていなかったか**：初期条件が一般的（一様＋ノイズ／
   単純な種）で、構造は後から出たか。
4. **入れていないものが出るか**：狙っていない随伴現象（音波・別の構造）も
   出るか。狙った物しか出ないなら台本を疑う。
5. **現実と数で合うか**：「似てる」でなく数で（循環が量子化／次元が本当に 3／
   周期や指数が理論値と一致）。
6. **頑健か**：初期条件・パラメータを変えても出るか。脆ければ作為。
7. **コードが結論を主張しているか、発見しているか**：答えを書き込んで
   いないか。測定して出しているか。

**7 つ全通過 → STATUS: GREEN（本物の創発と誇張なく言える）。**
一つでも欠ければ YELLOW/RED とし、何が足りないかを書く。
**GREEN 以外を「成功」と呼ばない。**

## 3. claim tier の定義

- **measured**：シミュレーション内で直接測った値（例：歳差角、循環の整数値）。
- **observed**：定性的に見えたパターン（数値の裏づけはこれから）。
- **interpretive**：概念への接続（解釈）。
- **analogy**：既知物理への構造的な類似（本物そのものではない）。
- **frontier**：未解決・投機的。

**誇張も卑下も禁止。** 各主張に必ず tier を付ける。
tier を上げるときは新しい測定で裏づける。

---

## 4. 忠実な場の規則（最重要の一線）

入れる「規則」は、**実在の物理法則か、無理のない最小の局所動力学**でなければ
ならない。**結果を出すために調整した生成器は禁止。** 規則が目標の構造を
符号化していたら、それは擬似。

- 良い例：GPE、波動方程式、反応拡散、TDGL、因果順序＋局所貼り合わせ。
- 悪い例：「ここに渦を置き、この軌道で回す」と結果を直接書く。

---

## 5. 再現性（擬似でない構造的保証）

- 各 `run.py` は乱数シードを固定し、`result.json` に主要測定値を書く。
- CI が core の単体テストと各実験の短縮版を回し、`result.json` の主要数値が
  期待範囲に収まるかを照合する。**CI が緑＝「掟つきで再現可能」。**
- 測定は `core/` の共有の物差しで行う。実験ごとに物差しを作り替えない。

## 6. 役割 — E / V / S / N / F / Q（二値 GREEN の置換）

各実験は**単一の役割**を持つ。`eXXX.py` と `AUDIT.md` の YAML ヘッダ（§8）に明記し、
`docs/TRUST_MAP.md` に集計する。

| 役割 | 名称 | 意味（判定基準） |
|---|---|---|
| **E** | Emergence | 忠実な法則＋最小条件から現象が勝手に出た。**7 監査＋第8監査を全通過**。狙っていない随伴現象も出る。**（旧 GREEN の中核）** |
| **V** | Validation | 教科書・既知の解析結果・独立文献と**数で一致**（例：Ra_c、π²/6、σ²=√(2D/k)）。**7＋8 監査を全通過**。V は E に**劣らない**——一致は硬い検証。 |
| **S** | Synthesis / Circuit | 設計した回路・目的をもって組んだ機構。**外部 oracle を含んでよいが必ず明示**（例：e025 の外部 oracle）。創発とは呼ばない。 |
| **N** | Negative | 負の結果（例：受動 phase-field は分裂しない e020）。正直な失敗は資産——隠さず残す。 |
| **F** | Frontier | 未解決・障害を定量化しただけ・機構実証のみ。GREEN と呼ばない。 |
| **Q** | Quarantine | **GREEN 剥奪**。主張とコードが不一致、または**第8監査に抵触（`target_encoded=true`）**の疑い。原因を書き、E/V へ戻すには修正＋再監査が要る。 |

**GREEN と呼べるのは E か V だけ**（7＋8 監査全通過）。S/N/F/Q は GREEN ではない。

## 7. 第8監査 — 結論の埋め込み検査（背骨）

> GPT 監査が見つけた一貫パターン：**失敗の多くは `if` 文で「死ね／協力しろ」と
> 書いたことではなく、結論と同型の因果を「方程式・初期条件・評価ゲート」に
> 埋め込んでいること。** これを機械的に外すのが第8監査である。

**第8監査（必須・全モジュール）**：
> **評価ゲートと初期条件（および正規化・スケール）は、結論と同型の因果を
> 符号化してはならない。**

各モジュールで次の4問に Yes/No＋理由で答える：

1. **初期条件が結論を先取りしていないか。** 例：e016 が初期サイズに √c₄ を入れて
   「size ∝ √c₄」を"出す"なら、それは測定でなく手入力。
2. **ゲートが `if` で結論そのものを書いていないか。** 例：e025 の `winding→gain`
   をゲート化していないか。
3. **ゲートが null／自明な系でも通ってしまわないか。** 例：e030 の `both_roles>0.9`
   が線形（a=1.0）でも 0.99 で通る＝分業を区別できていない。
4. **正規化・スケール・格子が答えを埋め込んでいないか。** 例：e010 が
   `spacing = L/√N` を入れて「間隔 ∝ 1/√N」を"出す"なら循環論。

**判定と処置**：
- どれかが Yes（埋め込みあり）→ **`target_encoded: true`**。
  - その主張が創発を騙るなら → **役割 Q（GREEN 剥奪）**。
  - 設計回路として意味があるなら → **役割 S**（oracle/回路と明示）。
- CI が YAML の `target_encoded` を機械チェックし、**`true` のものは E/V を名乗れない**。

## 8. YAML ヘッダ（§1 の 10 項目に加えて必須）

各 `eXXX.py` 冒頭の docstring と `AUDIT.md` 冒頭に、次の YAML を必ず置く：

```yaml
id: eXXX
role: E|V|S|N|F|Q
claim_tier: measured|observed|established|interpretive|analogy|frontier
evidence: "この主張を支える測定・既知一致（1文）"
target_encoded: false        # 第8監査：結論を初期条件/ゲート/正規化に埋め込んでいないか
known_match: "既知物理/文献との一致（V ならどの結果と何桁合うか）"
open_issues: ["未解決・床・次にやること"]
```

`claim_tier` に **`established`**（教科書・既知の解析結果との一致）を追加する
（V 役割の裏づけに使う）。

## 9. 三大原則（§0〜§8 と同格・全モジュール）

1. **「同じ数学」≠「同じもの」。** 同じ方程式が出ても「進化＝相分離」「地平線＝
   AB リング」等とは主張しない。**同じ定性事実が別基層でも出る**、に限る。
2. **誠実な床は隠さない。** 解像度限界・oracle・regime・finite-size・変動を必ず書く。
   通らないものを通す方向に**パラメータ・初期条件をいじらない**（それ自体が第8監査違反）。
3. **禁止語を「作った」と言わない。** `created life / consciousness / proved / perfect`
   等は主張しない（すべて analogy、docs のみ）。**ゲート名は物理量名**
   （`winding / bulk_amp / rotation / coop_fraction / mean_trait / relatedness` 等）。
