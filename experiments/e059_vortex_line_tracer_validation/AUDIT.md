# e059 3D渦線トレーサーの検証（H021キャンペーン PR-C・role V）

外部監査2件（GPT・別Claude、2026-07-16）が共に推奨した「新しい3D-native計測作業は渦線・閉ループ抽出の
一般計測器から始めるべき」という方向に沿って、`genesis/diagnostics/vortex_lines_3d.py`（プラケット磁束
から渦線を再構成する新規計測器）を実装し、既存の seeded vortex ring 実験（e003、無変更）で検証する。

## 新規計測器（vortex_lines_3d.py）の設計

`plaquette_ledger`（LT-3）が意図的に手を止めている「プラケット巻き数の投票（ballot）」から先の
「実際に線をトレースする」ステップを実装する。各格子立方体を制御体積として扱い、3方向のプラケット巻き数
から立方体の6面それぞれの符号付き磁束を計算（右手系：xy面法線=+z、yz面法線=+x、zx面法線=+y の巡回
規約）。ちょうど2面が貫通されている立方体は「clean」（線がその2面を通過）とみなし、面の中心同士を結ぶ。
このグラフを辿ることで閉ループ（渦線）を再構成する。

**接線領域の既知のギャップ処理**：曲がった線が格子座標軸にほぼ接する領域では、1面のみ貫通された
「dangling」立方体が生じる（離散化の既知の現象、marching cubes 系アルゴリズムや格子ゲージ理論の
渦線トレース法でも知られる標準的なケース）。これを最近傍再接続（`max_heal_distance`以内のみ、外部
レビュー対応、下記参照）で治癒し、治癒数・未治癒数・多重線曖昧性（>2面貫通、スキップ）を全て
診断値として報告する（隠蔽しない）。

## マージ前の外部AIレビュー第5ラウンド（CodeRabbit＋Codex）で発見・修正した6件

第4ラウンド後、それまでレート制限で待機していたCodeRabbitの初回レビューがようやく完了し、Codexの
追加レビューと同時に届いた。マージ前に検証・修正した：

1. **`run_valid`が無条件`True`だった**（CodeRabbit、Major）：測定パイプラインが1フレームも
   トレースせずに終わった場合（例外は出ないが空の実行）でも`run_valid=True`のままで、第4ラウンドで
   導入したばかりのCIゲート（`run_valid`ベースの終了コード）が効かなくなる欠陥だった。
   `len(frames) > 0`ベースに修正——`len(both) > 0`ではなく意図的に選択した：後者にすると
   マッチングヒューリスティックの成功率とCIの基本疎通確認（パイプラインが動いたか）を再結合させて
   しまい、第4ラウンドで苦労して切り離したはずの結合を再導入してしまう。
2. **コールドスタート時のマッチングに半径妥当性ゲートがなかった**（CodeRabbit）：
   `prev_matched_centroid`が未確立の最初のマッチでは、半径が最も近い候補を無条件で選んでおり、
   それが唯一の候補であればどれだけ半径がずれていても「マッチ」としていた——他の全箇所で確立した
   「妥当でない候補にフォールバックしない」規律から外れた非対称性。同じ50%帯ゲートを適用。
3. **到達不能な死んだコード**（CodeRabbit）：第4ラウンドでループの先頭に早期break
   （`if old is None: break`）を追加した際、ループ末尾に残っていた同等のチェックを消し忘れており、
   常に到達不能なコードとして残っていた。削除。
4. **AUDIT.mdの誤字**（CodeRabbit）：測定結果の説明で`n_frames_large_radius_diff`を2回記載して
   おり、2つ目は`n_frames_large_axial_diff`のはずだった。修正済み（本セクション上部）。
5. **マッチ失敗後も時間的連続性アンカーが古い位置を指したまま残っていた**（Codex）：ある
   フレームで妥当な候補が見つからず`matched=None`になっても、`prev_matched_centroid`は前回の
   位置のままリセットされず、次のフレームがその「古い」アンカーに最も近い妥当な候補（実際には
   リングが移動した後にたまたま近くに残っていた無関係な音波ループかもしれない）に誤って
   ロックオンし得た。マッチに失敗した時点でアンカーを`None`にリセットするよう修正（次のフレームは
   ゲート付きのコールドスタートロジックに戻る）。
6. **JSON成果物にGREEN/RED判定そのものが含まれていなかった**（Codex）：full runでは
   `evaluate()`がREDと判定しているにもかかわらず、書き出すJSONには`verdict`
   （insufficient_frames/agreement_measuredという粗い区別のみ）しか含まれておらず、このスクリプト
   を再実行せずにJSONだけを読む下流の監査・エビデンスツールは合否を復元できなかった。`main()`で
   `passed`/`checks`をJSONに永続化するよう修正。

回帰テスト2件を追加（`vortex.track_ring_cross_section`をmonkeypatchして常に`None`を返させ、
`run_valid`が実際に「測定なし」を検出することの検証；`main()`が書き出すJSONに`passed`/`checks`が
含まれることの検証）。修正後、`pytest tests/test_vortex_lines_3d.py
tests/test_e059_vortex_line_tracer_validation.py -q`で14件全通過を確認。

**修正がfull runの数値に与えた影響（正直な報告）**：数値は完全に変化なし（一致フレーム10/21、
半径・軸方向の乖離とも全フレームでゼロのまま）——この6件のバグは本データセットでは実際には
トリガーされなかった（ゼロフレーム実行は起きず、コールドスタートの初回マッチは既に妥当な範囲内
であり、マッチ失敗によるアンカーの陳腐化も本データセットの21サンプルでは問題を起こさなかった）が、
計測器と検証パイプラインの堅牢性はより正しくなった。

## マージ前の外部AIレビュー第4ラウンド（Codex）で発見・修正した3件——数値が大きく変わった重要な修正

1〜3回目の修正後、Codexからさらに3件の指摘が届いた。今回はこれまでと違い、**数値に大きな実質的
影響があった**（結果を良く見せる方向ではなく、むしろ「マッチしたと主張していたフレーム数」が
大幅に減った、より厳格で正直な方向）：

1. **CIの終了コードが統計的合否判定（`evaluate()`のGREEN/RED）に基づいていた**：`main()`が
   `return 0 if passed else 1`としており、既存のe057/e058が使う確立されたパターン
   （`return 0 if r["run_valid"] else 1`——測定が実行できたかどうかのみを見る、統計的な当落は
   見ない）から逸脱していた。本実験のfull runの`STATUS`は正直に`RED`であり、CIの`--quick`実行も
   決定論的に（乱数なし）毎回同じ閾値割れになるため、**このPRが最初に開かれてから今まで、
   GitHub Actions上のCIは実際には一度も緑になっていなかった**——ローカルの`pytest`実行のみで
   確認していたため、この事実に気づいていなかった（重大な見落とし）。`run_valid`ベースに修正し、
   e057/e058と同じパターンに統一。
2. **半径妥当性ゲート（第3ラウンド以前に導入）が、妥当な候補がゼロ件のときフィルタなしの全候補へ
   フォールバックしていた**：ゲートの意味そのものを無効化しており、「本来どの候補も妥当でない」
   フレームでも最近傍の（不適切な）ループを強制的に選び、マッチ済みとしてカウント——
   `matched_most_frames`と一致率指標の両方を水増ししていた。フォールバックを廃止し、妥当な候補が
   なければそのフレームは未マッチのまま正直に残すよう修正。
3. **`track_ring_cross_section`がNoneを返した時点（clean window終了）で、そのフレームがtrace/
   append済みの後にbreakしていた**：e003の終了規則（「これ以降は自身の読み取りを信用しない」）を
   厳密に再現するには、trace/append**する前に**breakする必要があった。範囲外の1フレームが
   `n_samples`・一致率計算に紛れ込んでいたバグを修正。

修正後、full run（21サンプル）を再実行した結果、**数値は大きく変化した**——正直な報告であり、
結果を良く見せるための事後調整ではない（むしろ逆方向、より厳しい結果になった）：
- マッチしたフレーム数が21中10（旧: 21）に減少——第2〜3ラウンドで「マッチ」と数えていたフレームの
  半分以上は、実は妥当性ゲートのフォールバックによる誤マッチだったことが判明。
- しかし残った10フレームでは、半径・軸方向位置とも**大きな乖離ゼロ**（`n_frames_large_radius_diff
  =0`、`n_frames_large_axial_diff=0`、共に100%一致）。
- これにより、計測器の限界の性質がより正確に特定された：**「見つけた時に間違う」のではなく
  「そもそも妥当な候補が見つかることが少ない」**（47.6%=10/21、`matched_most_frames`の80%閾値
  未達）。第1〜3ラウンドの「後半で半径が一致しなくなる」という解釈は、実は妥当性ゲートの
  フォールバックによる誤マッチの混入が原因だった——healing heuristicの根本的な脆弱性
  （チェーン端点単位ではなく個々の面単位で再接続している設計）自体は変わらず残っているが、
  その現れ方の理解がこの修正でより正確になった。
- STATUSは引き続きRED（`matched_most_frames`不通過）——判定基準がより正しくなったことで、
  かえって「妥当な候補が見つかった時は完璧に一致する」という、より強い正の結果も同時に
  明らかになった。

## マージ前の外部AIレビュー第3ラウンド（Codex）で発見・修正した3件のバグ

1回目・2回目の修正後、Codexからさらに3件の独立した指摘が届いた。マージ前に検証・修正した：

1. **同符号2面ペアをcleanとして扱っていた（`vortex_lines_3d.py`）**：2面貫通の判定が
   `abs(flux)==1`のみを要件としており、両面のフラックスが同符号（合計が非ゼロ＝発散違反）でも
   そのまま「clean」として接続していた。1本の線が入って出るなら2面のフラックスは符号が逆で
   相殺するはずであり、同符号ペアは物理的に不整合（信頼性ゲートが真の3枚目/4枚目の面をゼロ化した
   結果など）。`n_cubes_overloaded`に計上するよう修正（`>2`面貫通と同じ「曖昧、推測しない」
   バケット）。
2. **軸方向一致が判定基準に一度も使われていなかった（e059本体）**：`run()`は
   `new_axial_series`/`old_axial_series`の両方を記録していたが、`evaluate()`は半径の一致率
   （`radius_agrees_majority_of_frames`）しか見ておらず、`axial_diffs`という変数は宣言だけ
   されて実際には一度も使われていなかった。同じ半径だが軸方向位置が違う誤マッチでもSTATUS GREENに
   なり得た欠陥。半径チェックと全く同じ基準（Rの25%許容・70%多数決——事前に半径チェックと揃えて
   決定した値であり、結果を見てからの事後調整ではない）で軸方向一致チェックを追加。
3. **治癒に失敗した孤立ダングリング面がopen_pathsから完全に消えていた（`vortex_lines_3d.py`）**：
   `max_heal_distance`を超えて治癒できなかった面のうち、清潔立方体側の接続すら持たない孤立した
   ものは`n_unhealed_dangling`にはカウントされるが、`neighbors`グラフに一度も登録されないため
   `open_paths`の幾何情報から完全に欠落していた——「never silently dropped」という本モジュールの
   約束に反する。1点（`n_points=1`）のopen pathとして明示的に含めるよう修正。

回帰テスト2件を追加（`face_windings`をmonkeypatchして単一立方体の面フラックスを直接制御：
同符号ペアが`n_cubes_overloaded`に計上されること、孤立ダングリングが1点open pathとして
現れること）。修正後、`pytest tests/test_vortex_lines_3d.py tests/test_e059_vortex_line_tracer_validation.py -q`
で12件全通過を確認。

**修正がfull runの数値に与えた影響（正直な報告）**：半径一致の数値は完全に変化なし
（median radius diff = 7.206425349334724、前半70%・後半27%のまま）——この3件のバグは本
データセットでは実際にはトリガーされなかった（同符号2面ペアも孤立ダングリングも今回のリングの
軌跡上には現れなかった）が、計測器と検証基準はより正しくなった。軸方向一致チェックは今回
初めて追加され、85.7%（18/21フレーム、tolerance=Rの25%）でPASSした——半径は大きく外れても
軸方向（z位置）はよく一致しているフレームが多いことを裏付ける（新トレーサーが「同じz付近の
別の小さなループ」に誤ってロックオンしているケースが多いことと整合する）。STATUSは半径一致
未達のため引き続きRED——判定基準が完全になったことで数値がより厳密に検証されたのみで、
結果を良く見せるための変更ではない。

## マージ前の外部AIレビュー（Codex）で発見・修正した5件のバグ

本PR（未マージ時点）にCodexという外部AIレビューが自動で届き、5件の具体的な問題を指摘した。マージ前に
修正した——**マージ済みの数値を後から訂正したのではなく、下記が本実験の最初で唯一の公開測定値**:

1. **zx面の符号規約バグ（`vortex_lines_3d.py`）**：`_orientation`が("zx",(0,2))に対して計算する
   CCWループは、その軸0=x・軸1=zの並びから右手系の法線が実際には-yになるにもかかわらず、
   `_cube_faces`は+yとして扱っていた（xy面のx̂×ŷ=+ẑ、yz面のŷ×ẑ=+x̂は正しかったが、zx面だけ
   x̂×ẑ=-ŷという逆の巡回になることを見落としていた）。右手系の再導出と実データ（合成リング）で
   確認：修正前は56件の「発散違反」のうち32件がyz面とzx面が混在する立方体（曲がった線の接線領域で
   頻発）で発生しており、修正後はゼロになった（残り24件は本来1面しか貫通しない dangling 立方体のみで、
   これは不可避）。`n_divergence_violations`という誠実性診断値そのものが不正確だった。
2. **`seeded_structure`フラグ（experiment.yaml）**：e003と同じ`false`にしていたが、本実験は
   e003と異なり「自己形成」の主張を一切せず、検出対象のリング自体がそのままimprintされたものである
   ため、`true`が正確。e003自身は伝播という創発主張があるため`false`のままで正しい——両者は異なる
   意味を持つフィールドの正しい使い分け。
3. **open_paths のwalk順序バグ（`vortex_lines_3d.py`）**：開いた鎖（次数1の端点2つ＋次数2の内部
   ノード）を辿る際、内部ノードから歩き始めると`walk()`は片方向にしか進まないため、鎖が2つの断片に
   誤分割され得た。次数1の真の端点から先に歩くよう順序を修正（次数2ノードの走査は開いた鎖が
   全て消費された後に行う）。
4. **dangling再接続に距離上限がなかった（`vortex_lines_3d.py`）**：理論上、無関係な遠方の欠陥同士を
   橋渡しし、開いた/別々のパスを偽の閉ループにしてしまう恐れがあった。`max_heal_distance`パラメータ
   （デフォルト3.0格子単位、合成テストで観測される接線領域ギャップの実測1〜3単位を十分カバー）を
   追加し、それを超える対は未治癒のまま報告するよう変更。
5. **時間的連続性マッチングに半径妥当性ゲートがなかった（e059本体）**：前フレーム重心への最近傍のみで
   選んでいたため、実際のリングが別の場所に存在するのに近くの小さなノイズ由来ループへ誤って
   ロックオンし得た。old trackerの半径が利用可能なフレームでは、半径が妥当な範囲内
   （old半径の±50%）の候補に絞ってから最近傍マッチングするよう修正。

回帰テスト2件を追加（zx符号修正の検証：2面貫通の立方体は発散ゼロ、healing距離上限の検証：
上限を極端に小さくすると治癒件数がゼロになる）。修正後、`pytest tests/test_vortex_lines_3d.py
tests/test_e059_vortex_line_tracer_validation.py -q` で9件全通過を確認。

**修正がfull runの数値に与えた影響（正直な報告）**：修正後にfull run（21サンプル）を再実行した結果、
early clean window（前半）の一致率は70%（7/10）で変化なし、late clean window（後半）の一致率は
27%（3/11、修正前は36%）とやや悪化した。これは修正が計測器を「悪化させた」ということではない——
5件の分類変更（特にzx符号修正）が、既に存在していた既知の脆弱なhealing heuristicの成功/失敗
パターンを別のフレーム集合にシフトさせた結果である。この脆弱性自体（チェーン端点単位ではなく個々の
面単位で再接続している設計）は根本的には未解決のままで、次の反復候補として変わらず残る——数値を
良く見せるために閾値やアルゴリズムを事後的に調整することはしていない。

## e003（seeded vortex ring）での相互確認

e003と完全に同一の物理パラメータ・IC・伝播を再利用し、e003自身の`track_ring_cross_section`（子午面
2コア追跡器、無変更）と新しい`vortex_lines_3d`トレーサーを**同一のpsiフレーム**に両方適用して比較。

**e003自身の「clean window」自己終了規則を再利用**（新規に発明した基準ではない）：半径バンド逸脱
または軸方向の進行反転が検出された時点で比較を打ち切る——e003自身がその先の読み取りを信用していない
領域を、新計測器についても公平にテストしないため。

### 測定結果（full・L=64・R=10・観測窓 clean window 内で自己終了・21サンプル、外部レビュー第4ラウンド
修正後の最終結果）
- **「マッチした」と判定されたフレームは21中10**（47.6%）——`matched_most_frames`の80%閾値に未達。
  第3ラウンドまでの21中の残り（妥当性ゲートのフォールバックによる誤マッチ）を排除した後の、
  正直な「本当にマッチできたフレーム数」。

## 正直な評価（結果を見てから閾値を変えない規律）

1. **計測器自体は正しく動作する**：8件の合成テスト全てが、静的・完全対称なリングに対して計測器が
   正確であることを示している。これは結果を見た後に閾値を緩めたものではなく、最初から成立する
   独立した検証。
2. **実際のGPE場（音波・数値ノイズあり）では、healing heuristicに実データで発見した限界がある**：
   最近傍による dangling half-edge の再接続は、接線領域の局所ギャップを、正しい遠い接続の代わりに
   誤って近い自己閉合として「治癒」してしまうか、あるいは単に妥当な候補を全く生成できないことが
   多い。第4ラウンドの修正（妥当性ゲートのフォールバック廃止）により、この限界の性質がより正確に
   特定された：**計測器が「間違った答えを出す」のではなく「答えを出せないことが多い」**——前者は
   計測器の信頼性そのものへの疑義だが、後者は「保守的に振る舞い、自信がない時は何も主張しない」
   という、より望ましい失敗モードである。
3. **STATUS RED は正確な報告であり、隠蔽・後付けの閾値緩和はしていない**：`EXPECT`の閾値
   （tolerance=R の25%、マッチ率=80%）は最初に妥当な基準として設定し、実際の結果（マッチ率47.6%）
   に合わせて事後的に下げてはいない。「マッチした時は完璧」という強い正の結果と、「マッチする
   頻度が低い」という弱点を分けて報告することで、「どこまでなら信頼できるか」を正確に伝える。
4. **本実験はリングの自己形成を主張しない**：リングは e003 と同じく imprint された seeded
   structure。本実験が確立したのは「新しい一般用途の3D渦線計測器は、正しく設計されたシナリオ
   （静的合成データ・妥当な候補が見つかった実データフレーム）では正確に動作するが、healing
   heuristicの限界により実データでは候補を見つけられない頻度が無視できない」という計測器自体の
   妥当性と限界の両方。
